Chapter 01 · 序

先承认一件事 —
我们大多数人，只用了 30%。

Claude Code 是一个 agentic coding environment，不是一个聊天补全器。它能读文件、跑命令、改代码、连工具， autonomously work through problems。但绝大多数团队使用它的方式仍然停留在"问一句、改一行"。

问题不在工具，在使用层级。把本该是 hook 的东西写成提示词、把本该是 skill 的工作流复制粘贴到每个对话、 把本该交给 subagent 的探查塞进主上下文 —— 这是大部分团队"用不深"的真正原因。

这份手册不会讲所有功能。我们只覆盖真正决定使用深度的 13 件事 —— 全部基于 Anthropic 官方 Best Practices 与 Overview。

Claude's context window fills up fast, and performance degrades as it fills. Anthropic · Claude Code Best Practices

请把这句话刻在脑子里。本书后面所有的建议 —— 从 /clear 到 subagent、从 plan mode 到 worktree —— 都是为了一件事：把上下文当作最稀缺的资源来管理。

№ 0103 / 13 · THE THREE LAYERSUSE THE RIGHT LAYER
Chapter 04 · 核心工作流

Explore →
Plan → Code → Commit.

Anthropic 官方推荐的四阶段循环。把研究、规划、执行分离，避免一头扎进去解决错的问题。

i.

Explore

Plan Mode · Shift+Tab

让 Claude 先读，不写。回答你的问题。不让它修改文件。

read /src/auth and understand
how we handle sessions.
ii.

Plan

Plan Mode · Ctrl+G to edit

让 Claude 出方案。涉及哪些文件、流程是什么。Ctrl+G 在编辑器里直接改。

I want to add Google OAuth.
Create a plan.
iii.

Code

Default · Verify-as-you-go

切出 plan mode 让 Claude 实施。立刻跑测试 · 跑 lint · 跑编译。让它自己验证。

implement the plan. write tests
and run them, fix failures.
iv.

Commit

Default · Use git CLI

让 Claude 写 commit message 和 PR description。它比我们更耐心。

commit with a descriptive
message and open a PR.
什么时候不该 plan？ 改一行 typo、加个 log、重命名变量 —— 别 plan。Plan 是为了 "不确定方向 / 涉及多文件 / 不熟悉的代码"。一句话能描述的 diff 就跳过。 Source · Best Practices §Explore
大特性？让 Claude 反过来面试你。 用 AskUserQuestion 工具让它问 30 个问题， 覆盖技术 / UX / 边界条件 / 取舍，写完 SPEC.md 再新开一个会话实施。 Source · Best Practices §Interview
№ 0104 / 13 · WORKFLOWSEPARATE THINK FROM DO
Chapter 05 · 验证

给它一把
尺子。

"Give Claude a way to verify its work."
—— 官方文档原话：这是单一最高杠杆率的实践。

没有验证标准，Claude 会产出"看起来对"但实际不工作的代码。你会变成它的唯一反馈环 —— 每个错误都要消耗你的注意力。

Before · 模糊implement a function that validates email addresses
→
After · 可验证write a validateEmail function. cases: user@example.com → true, invalid → false. run the tests after implementing
Before · 模糊make the dashboard look better
→
After · 可验证[paste screenshot] implement this design. take a screenshot of the result, compare to original, list diffs and fix them
Before · 模糊the build is failing
→
After · 可验证build fails with [error]. fix it and verify the build succeeds. address root cause, don't suppress
If you can't verify it, don't ship it. Anthropic · Common Failure Patterns
在我们这个仓库里：后端用 mvn clean compile，前端用 pnpm build， 都已经封装进 verify skill。所有代码改动后，让 Claude 跑 /verify —— 不要靠人眼看 diff。 本仓库 · .claude/skills/verify/SKILL.md
№ 0105 / 13 · VERIFICATIONHIGHEST-LEVERAGE PRACTICE
Chapter 06 · CLAUDE.md 纪律

把 CLAUDE.md 当代码维护。
不当文档，当宪法。

CLAUDE.md 在每一次启动都会被注入到对话开头。它是 Claude 在你这个项目里的"先验知识"。 越短越好 —— 臃肿的 CLAUDE.md 会让真正重要的规则被淹没。

官方建议：对每一行问 —— "删了会不会让 Claude 犯错？" 如果不会，就删。 50-100 行是合理上限。详细内容用 @import 拆到 .claude/rules/。

✓ 应该写进 CLAUDE.md

  • Claude 无法猜到的 bash 命令
  • 与默认不同的代码风格
  • 测试运行器偏好（跑单测而不是全量）
  • 仓库礼仪：分支命名、PR 约定
  • 项目特有的架构决策
  • 开发环境怪癖、必需环境变量
  • 非显而易见的"坑区"

✕ 不要写进 CLAUDE.md

  • Claude 读代码就能知道的东西
  • 标准语言惯例（"用 ===")
  • 详细的 API 文档（链过去就行）
  • 会频繁变动的信息
  • 长篇解释或教程
  • 每个文件的逐项描述
  • "写干净的代码"这种废话
📁 全局
~/.claude/CLAUDE.md

个人偏好

跨所有项目生效 · 不进仓库 · 写你的 RTK、你的提交风格、你的个人癖好。

📁 项目
./CLAUDE.md

团队宪法

进 git · 全员共享 · 模块、架构、踩坑区。本仓库根目录已经做得很好 —— 看一遍就懂。

📁 私有
./CLAUDE.local.md

个人项目笔记

进 .gitignore · 只你看 · 写你和这个仓库的"小秘密"——临时密钥、还在尝试的工作流。

失败信号：如果 Claude 反复违反 CLAUDE.md 里的某条规则 —— 不是它笨，是规则被噪声淹没了。删 50% 内容比加"IMPORTANT"前缀更有效。 Source · Best Practices §CLAUDE.md
№ 0106 / 13 · CLAUDE.md DISCIPLINEPRUNE RUTHLESSLY
Chapter 07 · 上下文管理

上下文是最稀缺资源 —
花掉它，就回不来。

一个调试 session 能消耗几万 token。一个全仓搜索能再消耗几万。当窗口被填满，Claude 开始"忘记"先前的指令，犯更多错误。

最高频建议（来自经验最丰富的用户）：每个任务开新 session。任务间用 /clear 重置。 70% 用量时主动 compact。

命令用途
/clear无关任务之间·彻底重置上下文
/compact主动压缩历史·保留关键决策
/rewind Esc Esc回退到任意 checkpoint·撤销改动
/btw问一个不影响主线的小问题
Esc立刻中断·保留上下文
--continue继续上一次 session
--resume从历史 session 列表中选
/rename给 session 起名·跨天找回

Subagent ——
上下文的会计。

研究、审查、跨模块搜索 —— 这些任务会读几十个文件，全进主上下文就是灾难。 把它们交给 subagent：在独立窗口里跑，只把摘要回传给你。 

# 不要这样
read all auth files and tell me how OAuth works...
# 主上下文瞬间膨胀 50K tokens

# 应该这样
use a subagent to investigate how our auth
system handles token refresh. report findings.
№ 0107 / 13 · CONTEXTYOUR ONE CONSTRAINT
Chapter 08 · 提示词的精度

具体 ＝ 更少修正。
模糊 ＝ 你来当 QA。

Claude 能推断意图，但读不到你脑子里。引用具体文件、提约束、指向已有模式 —— 这三件事能把返工量砍掉一半。

例外：探索阶段反而要用模糊的 prompt。 "this file 里你会改进什么？" 能挖出你没想到要问的事。 Best Practices §Prompts
Beforeadd tests for foo.py
→
After · 限定范围write a test for foo.py covering the edge case where the user is logged out. avoid mocks.
Beforewhy does ExecutionFactory have such a weird api?
→
After · 指向来源look through ExecutionFactory's git history and summarize how its api came to be
Beforeadd a calendar widget
→
After · 引用模式look at HotDogWidget.php to understand patterns. follow it to build a calendar widget that lets the user select a month and paginate ...
Beforefix the login bug
→
After · 症状·位置·成功标准users report login fails after session timeout. check token refresh in src/auth/. write a failing test reproducing it, then fix it

№ 0108 / 13 · PROMPT PRECISIONSPECIFICITY WINS
Chapter 09 · 五种失败模式

如果你正在
感到吃力 —— 多半是这五件事之一。

i.

The Kitchen Sink Session

你开了个 session 修 bug A，顺手问了个 B，又回来改 A。三件事的上下文搅在一起。

Fix 无关任务之间一定 /clear。把每个 session 当成一个独立 ticket。
ii.

Correcting Over And Over

Claude 错了，你纠正；又错，你再纠正；第三次还错。上下文已经被失败的尝试污染。

Fix 纠正超过两次就 /clear。把你刚学到的东西写进新 prompt，从零开始。
iii.

The Over-Specified CLAUDE.md

你的 CLAUDE.md 已经 800 行。重要的规则被淹没在噪音里。

Fix 无情删减。"删了会不会让 Claude 犯错？"不会就删。重复执行的事改写成 hook。
iv.

The Trust-Then-Verify Gap

Claude 产出了"看起来对"的代码 —— 你没验证就 commit 了。第二天测试挂在 CI 上。

Fix always 提供验证手段（测试·脚本·截图）。如果你没法验证，就不该 ship。
v.

The Infinite Exploration

你让 Claude "看看这个项目"。它读了 200 个文件。你的上下文已经满了，还没开始干活。

Fix 研究类任务用 subagent。给探索一个明确边界："3 个文件以内·5 分钟以内·回答这一个问题"。
№ 0109 / 13 · FAILURE PATTERNSRECOGNIZE THEM EARLY
Chapter 10 · 横向扩展

一个 Claude 不够 —
开四个。

前面所有章节都假设：一个人、一个 Claude、一段对话。但 Claude Code 是水平扩展的工具。

Headless

非交互 模式

塞进 CI、pre-commit hook、shell 脚本。给固定输入，拿结构化输出。

# 一行命令
claude -p "explain this"

# 结构化输出
claude -p "list endpoints" \
  --output-format json
Worktree

并行 session

用 git worktree 把同一个仓库 checkout 到 N 个目录，每个跑一个 Claude，互不污染。

git worktree add ../wt-a feature-a
git worktree add ../wt-b feature-b
# 两个 Claude 同时干
Fan-Out

批量 处理

大型迁移、批量审计 —— 用 for 循环跑 N 次 claude -p，每次处理一个文件。

for f in $(cat files.txt); do
  claude -p "migrate $f. OK or FAIL" \
    --allowedTools "Edit"
done

Writer
×
Reviewer.

Session A 写代码 → Session B（独立上下文 · 没看过 A 的实现细节）做 code review → 把 review 反馈喂回 A 修。

为什么有效？一个 Claude 不会对自己刚写的代码"挑刺"—— 有偏见。两个 Claude 互相约束， 等于免费多了一道质量门禁。

类似套路：一个写测试，另一个写代码让测试过。或者一个实现 backend，一个实现 frontend， 主 agent 协调集成。 Best Practices §Multi-Claude

所有的模式 ——
都不是定律。

有时候你应该让上下文累积，因为你正在深入一个复杂问题，历史本身有价值。 有时候你应该跳过 plan 模式，让 Claude 直接试错。 有时候一个模糊的 prompt 恰恰是对的 —— 你想先看看 Claude 怎么理解，再去约束。

留意什么有效。Claude 产出漂亮代码时 —— 想想你做了什么。Claude 卡住时 —— 想想为什么。 上下文太杂？prompt 太空？任务太大？时间久了，你会培养出一份这本手册抓不到的直觉。
